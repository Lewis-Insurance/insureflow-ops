1What this page is

The front door to the system that runs the agency.

This is the staff authentication gate for the Lewis Insurance Agency OS (the InsureFlow Ops platform). It's the screen the team starts every workday on — no public signups; access is email + password, then gated post-auth on is_staff + active status. Three jobs: get an authorized person in fast, signal a real and secure product, and carry the brand as the first thing anyone sees. It's the most-seen screen in the build, so it sets the tone for everything else.

The "two Sign In buttons" — root cause & fix

The form is a shadcn <Tabs>. Because ENABLE_SIGNUP defaults false, the <TabsList> still renders with one tab — a "Sign In" pill floating above the fields — while the submit button at the bottom is also "Sign In." That's the redundant pair in your screenshot. Fix: render the plain form and skip the tab strip entirely when signup is disabled.

// Auth.tsx — only wrap in Tabs when signup is actually enabled (lines ~182-186)
- <TabsList className={`grid w-full ${ENABLE_SIGNUP ? 'grid-cols-2' : 'grid-cols-1'}`}>
-   <TabsTrigger value="signin">Sign In</TabsTrigger>
- </TabsList>
+ {ENABLE_SIGNUP && (
+   <TabsList className="grid w-full grid-cols-2">…two triggers…</TabsList>
+ )}
// Result: exactly one primary "Sign in" on the page when signup is off (the default).

7Layout & responsive

  • ≥1024px — split: brand panel 55% left, auth column 45% right.
  • 768–1023px — brand collapses to a ~30vh top band (logo + headline over the tide); form centered below.
  • <768px — slim ~108px navy header (logo + thin animated wave strip); form full-width; 50px tap targets.

The tide (the signature — don't over-build it)

Three layered SVG wave paths (#11314F / #0C2742 / #08203A) animated by CSS translateX; one orange horizon line at 70% opacity that pulses; one breathing radial orange glow; a 3.5% grain overlay (feTurbulence); a faint sail watermark. That's the whole recipe. No canvas, no three.js, no parallax, no third accent color — the restraint is the brand. Build it as inline SVG + CSS keyframes (GPU-friendly).

Logo polish before it propagates (it's raster/AI-generated today)

  • Build a dedicated simplified favicon — the interior waves blur below ~32px. Reduce to shield + two wave strokes + dot.
  • Resolve the floating orange dot — anchor it as a deliberate "rising signal" in the shield gap, or drop it.
  • Pick one shield behavior — the horizontal lockup lets waves spill past (and tangent) the shield edge; stacked/reversed contain them. Containment sells "protection." Unify it.
  • Even out wave stroke weights and leave a sliver of clearance from the shield outline (kills tangents).
  • Ship the full set when vectorized: full-color, reversed, icon-only (have these) + all-white + all-navy monos + the favicon.

Inputs

Persistent visible label above each field (never placeholder-only). Rest border #D8E0EA → hover #C2CDDA → focus navy border + orange glow ring. Invalid: error border + inline message with icon, wired via aria-describedby. Restyle :-webkit-autofill so it stays white-bg / ink-text (don't let it turn yellow).

Password field

Trailing show/hide eye (44px target, toggles type + aria-pressed); inline "Caps Lock is on" hint via getModifierState('CapsLock').

Primary button

Brand Navy bg + white text (not orange — fails AA, see §10). Hover → darken + lift + shadow; active → scale .99; loading → spinner + "Signing in…", disabled; success → morph to check + "Signed in"; then route.

Google SSO — removed for v1

Per your call, the login is password-only at launch (matches the app today): no SSO button, no "or" divider. Wiring notes are retained in §11 for when you add it.

Form-level error banner

Top of form, role="alert" aria-live="polite", icon + message. Used for credential/locked/network/not-authorized cases.

Security note on error copy

Your current handleSignIn deliberately collapses bad-email and bad-password into one generic message (anti-enumeration). Keep that for the bad-credentials case. "Not authorized / locked / network" are distinct UX states you can add only where the Supabase response actually distinguishes them — otherwise fall back to generic. Don't leak which accounts exist.

Two things this proves

  • Primary button stays navy, not orange. Orange + white text is 2.95:1 (fails). If brand insists on an orange CTA to echo marketing, the only AA-safe combo is orange bg + navy text (#0A2038). Recommend navy bg.
  • Footer uses Fog, not the lighter Haze (#9AA7B6 was 2.45:1 — failed; corrected in the prototype).

Also required: visible :focus-visible ring on every interactive element; logical tab order; Enter submits; email autofocus; the rest-state field border (1.33:1) is decorative — fields are identified by label + fill + the high-contrast focus state (common AA-passing pattern); target Lighthouse a11y ≥ 95.

11How to build it in this stack

1 · Fonts (Vite, not next/font)

<!-- index.html <head> -->
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;500;600;700&family=Plus+Jakarta+Sans:wght@600;700;800&display=swap" rel="stylesheet">

Then set the families in tailwind.config.ts (fontFamily.sans = ['Inter',…], fontFamily.display = ['Plus Jakarta Sans',…]). Self-host later if you want zero third-party requests.

2 · Tokens

Add the §5 HSL values as CSS vars in src/index.css (:root) and register them in tailwind.config.ts under colors, following the existing hsl(var(--x)) pattern.

3 · Refactor Auth.tsx

Build the split layout; drop the Card; apply the Tabs fix (§3). Reuse the existing signInData state, validateSignInForm, and handleSignIn untouched — only the markup/classes change. shadcn Input/Label/Button/Alert can stay (restyled via tokens) or be replaced with token-styled elements.

4 · Google SSO — deferred (removed for v1)

// Only after the Google provider is enabled in Supabase Auth
// (Dashboard → Authentication → Providers → Google) + redirect URLs set:
await supabase.auth.signInWithOAuth({ provider: 'google',
  options: { redirectTo: `${location.origin}/` } });
// Removed for v1. When you add it: enable the provider, render the button, then call this.

5 · Forgot-password — no route exists

Either (a) add a /reset route using supabase.auth.resetPasswordForEmail(), or (b) hide the link for v1. Don't ship a link to nowhere.

6 · Logo

The InsureFlow mark exists but is raster/AI-generated. The prototype ships a clean SVG interpretation of the shield-and-waves (reversed + full-color) you can lift. For production, get the official logo professionally vectorized to /public/ (full-color, reversed, icon-only, mono-white, mono-navy) and build the simplified favicon (§8); swap the favicon set in index.html. The prototype auto-loads insureflow-logo-color.png / insureflow-logo-reversed.png from its folder if present (drop in transparent-bg exports); otherwise it shows the vector stand-in.

Lovable artifacts

lovable-tagger is dev-only in vite.config.ts — fine to leave. No "Pages" badge exists in index.html; nothing to remove.

12Acceptance criteria

  • npm run typecheck + lint + npm run build pass.
  • Exactly one primary "Sign in"; the phantom Tabs button is gone.
  • Split at ≥1024, top-band 768–1023, slim header <768.
  • Every input has a visible persistent label; orange focus-visible ring on every interactive element.
  • States implemented: focus/hover, show/hide + caps-lock, loading/disabled/success, field errors, form banner (bad-creds / not-authorized / locked / network).
  • prefers-reduced-motion freezes waves + glow.
  • WCAG AA verified (navy button, deep-orange links); autofill keeps field styling.
  • Enter submits; email autofocus; logical tab order; Lighthouse a11y ≥ 95.
  • Supabase auth, is_staff, RLS untouched.
  • Prototype-only chrome (the demo "preview states" pill) is not ported to production.
