## 1. The ask

Lewis & Lewis Insurance generates ACORD forms constantly. ACORD 25 (Certificate of Insurance) is the most common — proof of liability coverage for a client's certificate holders (landlords, lenders, general contractors, fleet management companies, municipalities, etc.). Forms 125/126/140 are used for quoting/binding through carriers but are explicitly **out of full-support scope** for this module (they need long-form manual answers to industry-specific questions that don't belong in structured CRM data) — the module should be architected so they *could* be added later without a rewrite, but only ACORD 25 needs to actually work end to end.

The module has three pieces:

- **Master COI** — a new panel on a specific customer's record page showing every field an ACORD 25 needs (carrier names, policy numbers, effective/expiration dates, coverage limits, NAIC codes), pulled from that customer's policies, editable and easy to double-check before a cert goes out.
- **Additional Insureds** — a universal, reusable directory of certificate holders (companies, individuals, other interests) that any customer can attach to a cert, with duplicate detection so "Enterprise Fleet Management" doesn't get created five times under five different customers.
- **ACORD Forms** (the generation module) — deliberately simple once the above is populated: Customer → Policies (checkboxes) → remarks/notes field → Additional Insured → Generate → downloadable/printable PDF, saved automatically into that customer's Documents tab.

Constraints Brian called out up front: Policies under the Customer page need updating to support this without breaking current usage; Master COI needs to be easy to read/edit and route correctly everywhere; Additional Insureds needs real duplicate detection, not just a flat list; the generated PDF needs to look clean and be genuinely ACORD 25 (not a lookalike), suitable to email a client; every generated COI needs to land in that customer's Documents tab automatically.

## 2. Decisions already locked in (this session)

Brian reviewed the trade-offs and made these five calls. Fable's plan should build on top of them, not re-open them:

1. **Reuse the existing ACORD engine.** The Master COI / Additional Insureds system should be a data layer that feeds the existing `/acord-forms` + `/acord-templates` rendering pipeline (`FormManagement.tsx`, `AcordTemplates.tsx`, `pdfFiller.ts`) rather than a new parallel PDF-filling system. **Important caveat surfaced by the deep dive (Section 3.1): this pipeline currently cannot actually produce an ACORD 25 at all** (no template onboarded, and its ingestion path actively rejects the XFA-hybrid PDFs that real-world ACORD 25s commonly are). "Reuse" therefore means reusing and fixing the engine, not just pointing at something that already works.

2. **Extend the `policies` table** for Master COI coverage-line data (limits, NAIC, insurer letter assignment, ADDL INSD/waiver flags per line) rather than building a fully separate parallel data model. Section 3.2 documents exactly what already exists on `policies` and its ~25 child tables vs. what's missing, so this extension is additive and targeted rather than a guess.

3. **Reuse the relationship-graph fuzzy-matching engine** (pg_trgm trigram search + staged suggestions + human-confirm pattern, built 2026-06-28/29 for account dedup) for Additional Insureds duplicate detection. Section 3.3 is explicit about what's a true drop-in reuse (`normalize_entity_name`, the `duplicate_groups` table shape) vs. what's architecturally-similar-but-must-be-cloned (search RPC, suggestion generation, merge engine — all currently hardcoded to `accounts.id`).

4. **Snapshot issued COIs.** An issued certificate is an immutable, versioned historical record — editing a policy later must never silently change a cert already sent to a holder. This matches Brian's reference doc's strong recommendation (its Section 4.4) and is a real gap in both existing COI-adjacent systems today (see 3.1.7).

5. **Track real Additional-Insured endorsement status**, rather than defaulting every cert to `ADDL INSD = Y`. This is the single highest-stakes correctness/E&O issue flagged in the reference doc (its Section 7): marking a holder as an additional insured is a legal assertion that an endorsement actually exists on the underlying policy. The repo already has a real, if incomplete, home for this: `policy_cgl_additional_insureds` / `policy_umbrella_additional_insureds` carry an `ai_type` field today (Section 3.2) — but only for GL and Umbrella, not Auto or Property, and there's no concept of "requested but not yet endorsed" as a distinct state from "confirmed endorsed." The new module should make this state explicit and visible, not just a checkbox.

## 3. Current repo reality (verified against source, not against CLAUDE.md)

### 3.1 The existing ACORD Forms engine — what's real, what's not

Two separate, non-integrated systems already claim to handle "COI" in this repo, and the live UI entry point today points at the weaker of the two:

**System A — the "ACORD Form Automation Suite"** (`supabase/migrations/20251218204626_acord_form_automation_suite.sql`, dated Dec 2025, not Dec 2024 as `CLAUDE.md`'s changelog implies). Schema: `acord_templates` (`form_number`, `form_name`, `version`, `is_current`, `pdf_type` default `'acroform'`, `pdf_template_url`, JSONB `field_inventory`/`field_schema`/`section_definitions`/`validation_rules`/`signature_anchors`/`repeater_configs`) and `acord_forms` (`account_id`, `template_id`, `field_values` JSONB, `pdf_url`, `cloned_from` self-FK, `signature_status`, `row_version`), plus `acord_form_sections` and `acord_field_audit`. Routes: `/acord-templates` → `AcordTemplates.tsx`, `/acord-forms` → `FormManagement.tsx`, `/acord-forms/:id/edit` → `AcordFormEdit.tsx` (all wired in `src/App.tsx`).

  - **No ACORD 25 template exists in the system at all** — no template row, no field catalog, no blank PDF asset anywhere in the repo. Form 25 exists only as a metadata stub in `src/types/acord.ts:477`. This system has never actually produced an ACORD 25.
  - Fill mechanics live in `src/lib/acord/pdfFiller.ts` (654 lines), client-side pdf-lib. It does **not** set `NeedAppearances` (zero matches repo-wide) — instead it calls `form.updateFieldAppearances(font)` then `form.flatten()`. This is a legitimate alternative to `NeedAppearances` (it bakes appearances permanently rather than depending on the viewer to regenerate them), but it still needs the same visual verification the reference doc calls for (Section 5.6 below) — flattening has its own failure modes if done before appearances are generated correctly.
  - **No `[0]`-suffix XFA-hybrid handling exists, and `src/lib/acord/templateIngestion.ts` (~lines 88–96) actively rejects XFA-hybrid PDFs on ingestion.** This is a concrete blocker: most carrier-issued ACORD forms, including ACORD 25, are commonly distributed as XFA-hybrid PDFs. Onboarding a real ACORD 25 template will likely hit this rejection immediately unless it's fixed.
  - `AcordFormEdit.tsx`'s `handlePullFromAccount()` (~lines 268–311) maps roughly 14 keys from only 7 `accounts` columns (name/address/phone/email) and **pulls zero policy data**. Since a COI's substance is policy data (carriers, policy numbers, dates, limits), this pull is not usable as-is for Master COI.
  - **No base+overlay / "fill once, swap the holder" mechanism exists.** Every generation (`useAcordForms.ts` `generatePdf()`, ~lines 279–345) re-fetches the full template and re-fills every field from scratch. "Clone Form" only duplicates the `field_values` JSON blob; it still requires a full fresh PDF render. This means the Master-COI-plus-holder-overlay plug-and-play model Brian wants is **not a native capability today** — it needs to be built on top.
  - Document import (`DocumentImportModal.tsx` → edge function `acord-document-extractor-v2`) and audit history (`acord_field_audit`, surfaced via `acord_audit_history` view) are real and functional. eSignature is real too: `supabase/functions/esign-create-request/index.ts` makes genuine authenticated calls to Dropbox Sign's `hellosign.com/v3` API — this is the one CLAUDE.md claim about this system that fully holds up.
  - Dead/orphaned code to be aware of: `supabase/functions/pdf-generation-worker/index.ts` (442 lines, its own independent fill implementation) has **zero call sites from the frontend** — despite CLAUDE.md describing it as the generation path. The `acord_generation_jobs` queue table is likewise unwired. There's also a storage-bucket naming mismatch: the live path writes to the `documents` bucket under `acord-forms/...` key prefixes, while the orphaned worker function targets a literal `acord-forms` bucket that no migration ever creates.

**System B — the legacy jsPDF-based COI generator**, and **this is what the live UI actually calls today**: the customer record page's overflow menu item "New certificate" (`src/pages/CustomerDetail.tsx:429`) navigates to `/coi-generator?accountId=...`, i.e. `src/pages/COIGenerator.tsx` / `useCOIGeneration.ts` / `src/lib/pdfGenerator.ts`, backed by the `certificates_of_insurance` table (`supabase/migrations/20251011003211_...sql:59-80`: `account_id`, a **single** `policy_id` FK, `certificate_number` auto-generated `COI-YYYY-NNNNN`, `certificate_holder_name`/`certificate_holder_address` JSONB, `effective_date`/`expiration_date`, `coverage_details` JSONB, `additional_insureds` JSONB, `special_provisions`, `document_url`, a `status` workflow draft→pending_review→approved→sent, `ai_generated`, `generated_by`/`approved_by`, `sent_at`) and storage bucket `certificates`. This system is operationally mature (numbering, status workflow, audit fields, RLS) but **renders a hand-drawn jsPDF layout — cosmetically COI-like, not an actual ACORD 25 form fill.** It's also architecturally single-policy-per-cert (`policy_id` is one FK, not a set), which doesn't match ACORD 25's multi-coverage-line-per-cert reality.

**Implication for Fable's plan:** "reuse the existing engine" (Decision 1) most plausibly means building the real ACORD 25 template into System A (fixing the XFA rejection bug first), building the Master COI data layer to feed it, and making an explicit call on System B — retire it, redirect its UI entry point to the new flow, or keep it running in parallel for some transition period. Leaving both live and unreconciled, as today, means two different "New Certificate" concepts coexist and the UI's own overflow menu currently points at the one that isn't a real ACORD form.

### 3.2 Policies, accounts, and Documents — what data already exists

**`policies` table** (base created `supabase/migrations/20250908032636_...sql:64-79`: `id, org_id, account_id, carrier_id, policy_number, line_of_business, effective_date, expiration_date, premium, payment_type, status, created_at, updated_at, deleted_at`) has grown into a hybrid structure via ~13 later migrations — not the simple single-row-per-policy table `CLAUDE.md` describes:

- Free-text `carrier`, `coverage` JSONB, billing fields added `20250908182341_...sql:101-107`.
- Five line-specific JSONB blobs added Dec 2025: `wc_details`, `bap_details` (commercial auto), `cgl_details` (commercial GL — this is where **GL limits live today: Each Occurrence, General Aggregate, Products-Comp/Op Agg, Personal & Adv Injury — as JSONB only, no typed columns**), `property_details`, `umbrella_details`.
- ~25 normalized child tables FK'd to `policies.id` hold structured per-line detail: `policy_wc_*`, `policy_bap_*` (vehicles/drivers/coverages), `policy_cgl_*` (locations, additional insureds, endorsements), `policy_property_*`, `policy_umbrella_*`.
- `policy_cgl_additional_insureds` / `policy_umbrella_additional_insureds` carry typed `ai_type` and `waiver_of_subrogation` fields — **but only for GL and Umbrella lines; Auto and Property have no equivalent today.**
- Auto CSL and WC E.L. limits currently only appear as typed columns inside `policy_umbrella_underlying` (which describes *referenced* underlying policies for umbrella purposes) — **not as first-class fields on the Auto/WC policy's own tables.**
- NAIC exists in three places: `carriers.naic`, `mgas.naic` (a separate table for MGAs/wholesalers), and `policies.carrier_naic`. A `carriers` table exists, but `policies.carrier` remains deliberately free text — a backfill migration (`20260628014122_...sql:27`) explicitly never overwrites it even while populating a parallel `carrier_id` FK via alias mapping. `AddPolicyModal.tsx` still writes free-text `carrier` via a datalist, not a normalized picker.
- **Nothing in the schema has a home for**: Certificate Holder as a concept (the closest is `policy_cgl_additional_interests.interest_type` having a `certificate_holder` enum value shared with mortgagees — not a purpose-built holder entity), Description of Operations free text, or a mailing address on `accounts` distinct from its one physical address block (no `dba` on `accounts` either — that only exists on `businesses` and `policies`).

**`AddPolicyModal.tsx`** (zod schema at lines 22–32) collects `policy_number`, `carrier` (free text), `line_of_business`, `premium`, `billing_frequency`, `policy_term`, `billing_method`, `effective_date`, `expiration_date`, `status` — no NAIC, no coverage limits, no certificate holder. This is the concrete gap Master COI needs to close, either by extending this form or by giving Master COI its own richer editing surface for the fields AddPolicyModal doesn't capture.

**`DuplicatePolicyDialog.tsx`** is reactive only — it fires after a Postgres unique-constraint violation on `policy_number` (`AddPolicyModal.tsx` ~lines 316–357), not proactive fuzzy matching. Any Master COI logic that reads across a customer's policies should not assume near-duplicate policies (e.g., a renewal with a slightly different number) are deduplicated anywhere today.

**`CustomerDetail.tsx`** (589 lines) confirms the June 30, 2026 rework to always-visible stacked panels via an explicit code comment (lines 103–105): `SECTION_IDS = ['contact', 'policies', 'relationships', 'documents', 'notes', 'activity']`, each rendered as an independently-built `<section id="...">` block — there's no shared panel-wrapper component to extend, each panel (e.g. `CustomerPoliciesSection`, `CustomerDocumentsSection`) is its own component imported into the page. A new Master COI panel should follow this exact convention: a standalone component, a new `<section id="master-coi">` placed logically after `policies` (since it's built from policy data) and before/alongside `documents` (since it produces documents), and an added entry in `SECTION_IDS`. The hero action bar (~lines 320–459) already has an overflow `DropdownMenu` containing both "New certificate" (line 429, → legacy `/coi-generator`) and "Add policy" (line 432) — meaning the entry points for both Master COI and ACORD Forms generation are already anticipated in the UI's structure; the "New certificate" item is the natural place to repoint once the new module exists (see open question in Section 4).

**Documents.** Rendered by `DocumentCollectionBoard.tsx` and `CustomerDocumentsSection.tsx` (via `useDocumentManager`). Storage bucket `documents` (confirmed via `.storage.from('documents')` calls in `AddPolicyModal.tsx` and `UploadDocModal.tsx`; the bucket itself isn't provisioned by a tracked migration). The `documents` table (`20250908032636_...sql:173-187`: `id, account_id, policy_id, kind, filename, storage_path, file_size, mime_type, pii_level`) was extended with a `category` enum lacking a COI value, but a **newer `document_type` TEXT column with a CHECK constraint already includes `'coi'`** as a legal value (`20251204000003_add_document_classification.sql:11-36`, alongside policy/quote/dec_page/endorsement/claim_form/bill/loss_run/application/renewal/cancellation/binder/certificate/inspection/unknown). **This means "save the generated COI into Documents" already has a ready home** — insert with `document_type = 'coi'`, `account_id` set, and `policy_id` optionally pointed at a representative underlying policy — no schema change needed for this specific piece.

### 3.3 The relationship-graph dedup engine — what's truly reusable

The account-dedup system built 2026-06-28/29 is a genuinely good pattern, but almost none of its SQL is directly callable against a new entity type — it's hardcoded to `accounts.id` throughout, not polymorphic:

- **`search_accounts`** (`20260629250000_relgraph_v2_search_owned_rollup.sql:12-57`) matches `accounts.name`/`goes_by`/`account_aliases.alias`/`email`/`phone` via ILIKE plus `similarity(a.name, term) > 0.3` trigram fallback (min 3 chars), backed by GIN trigram indexes. `pg_trgm` is already enabled (`20250908050148_...sql:2`). The function body is `from public.accounts a` — hardcoded, not parameterized.
- **`generate_relationship_suggestions()`** (`20260629230000_relgraph_v2_suggester_rebuild.sql:24-178`) runs eight distinct signal CTEs (business-email-name, surname-business gated by a corroborating shared contact, shared-owner business pairs, shared-contact business pairs, spouse-name matching, address+surname with asymmetric ownership, shared-address households, catch-all shared-contact), ranks one best candidate per pair, and is `security definer`/staff-gated. All eight signals are meaningful prior art for an Additional-Insureds matching function but would need to be re-authored against a new table.
- **`account_relationship_suggestions`** (`from_account`/`to_account` FKs, `rel_type`, `signal`, `confidence`, `status`, `reviewed_by/at`) has a working **confirm** RPC (`confirm_relationship_suggestion`) but **no reject/dismiss RPC** despite the status column supporting it — worth noting as a small existing gap, not just a new-module concern.
- **`duplicate_groups`** (`20250908040318_...sql:13-23`) is genuinely generic in shape (`entity_type text`, `entity_ids uuid[]`) and is a **separate, older mechanism** from `generate_relationship_suggestions` — populated by a one-time backfill script, not the live suggester. However, every current reader RPC (`list_duplicate_groups_for_review`) hardcodes `entity_type='accounts'` and joins `accounts` directly, so the table shape is reusable as-is for Additional Insureds (`entity_type='additional_insureds'`) but the reader RPC needs a variant.
- **The merge engine** (`_do_account_merge`, `20260629240000_relgraph_v2_merge_consolidation.sql:25-266`, reached via `merge_accounts_manual`/`preview_merge`/`relgraph_merge_duplicate_group`) has a genuinely generic core — it introspects `pg_constraint` for every FK pointing at `accounts` and reparents automatically — but is wrapped in accounts-specific scalar-merge logic, a hardcoded safe-delete table allowlist, and accounts-specific guards (`assert_mergeable` for TIN/DOB/FEIN conflicts). **The pattern is reusable, the code is not** — a new entity needs its own merge function cloning this shape.
- **`normalize_entity_name`** (`20260629190000_import_resolve_account.sql:28-42`) is pure, generic string normalization (lowercase, `&`→`and`, strip non-alphanumerics, deliberately keeps LLC/Inc suffixes) with zero account-specific logic. **This is a true drop-in reuse** for Additional Insured name normalization.
- **`LinkAccountDrawer.tsx`** (250ms-debounced typeahead calling `useAccountSearch()` → `search_accounts`) is the UI template to fork for an "Add Additional Insured" live-duplicate-check field; no separate pattern for this exists elsewhere in the repo.

Net: budget for a parallel `additional_insureds` table, a `search_additional_insureds` RPC (clone the ILIKE/trigram approach), a suggestions mechanism (either a new table mirroring `account_relationship_suggestions`'s shape, or a carefully-scoped polymorphic rework — the former is lower-risk), reuse of `duplicate_groups` with a new `entity_type` value plus a new reader RPC, and a merge function cloning `_do_account_merge`'s FK-introspection approach scoped to the new table.

### 3.4 Design system requirements (Calm Command — binding, not optional)

Full rules live in `design-system/*.md`; the ones most load-bearing for this module:

- **Exactly one lime primary action per surface.** For the ACORD Forms generation flow, that's **Generate**; Save/Preview/Add-Additional-Insured must be olive/ghost/text-tertiary, never a second lime fill.
- **PII masking is mandatory, not a nice-to-have, and applies to previews and exports, not just live fields.** `design-system/constitution.md` line 57 and `surface-map.md` line 52 both call this out specifically for ACORD-style documents: "ACORD apps surface DOB and DLN. They render masked in the preview and are excluded from exports unless the document legally requires the full value and the user confirms." `src/components/cc/mask.ts` already exports `maskTaxId`/`maskDln`/`maskDob` — reuse these directly rather than reimplementing.
- **Carriers are name chips, never color-coded** (constitution.md line 52, reinforced in anti-patterns.md line 18) — directly rules out any idea of visually distinguishing carriers by color anywhere in Master COI or the ACORD picker.
- **Tabular figures; money, dates, policy numbers, and client names never truncate**, explicitly including comparison-grid-style layouts (constitution.md line 56) — applies directly to the policy-checkbox list in the ACORD Forms module and to every limit/date field in Master COI.
- **No em dashes or en dashes in interface copy** (constitution.md line 59).
- **Archetype mapping** (`design-system/surface-map.md`): Master COI is a **panel/tab inside the existing Record Command page** (the customer record), not a new page — it belongs in the same anchored-section pattern `CustomerDetail.tsx` already uses. Additional Insureds is an **Index/List** archetype (page title, one lime "Add" action top-right, dense uniform-row table, dedup signals as labeled pills, not color alone). ACORD Forms generation maps almost exactly onto the doc's **Document Production** archetype (lines 47–53): "a source-data selector on the left (policies, certificate holder, description of operations), a live document preview on the right, one primary action (Generate), and an issuance log beneath" — this is close to a literal spec for the module Brian described.
- **`src/components/cc/DateField.tsx`** (added June 30, 2026) is a keyboard-first ISO-date input (parses `MM/DD/YYYY`, `M/D/YY`, `YYYY-MM-DD`, `MMDDYYYY`, rejects impossible dates, snaps back on invalid entry, optional calendar popover) — the correct primitive for every effective/expiration date pair in Master COI and the ACORD picker, rather than a native date input.
- **`CustomerDetail.tsx`'s hero pattern in practice**: one lime `data-primary` action, a secondary action, outline buttons, and a three-dot overflow menu for lower-frequency actions (where "New certificate" and "Add policy" already live) — confirming both new entry points slot into the existing overflow menu rather than needing new top-level buttons.

## 4. Open questions surfaced by the deep dive (not yet decided — Fable should resolve these)

1. **What happens to the legacy `COIGenerator.tsx` / `certificates_of_insurance` system?** It's what the live UI currently calls, it's operationally mature (numbering, status workflow, RLS), but it doesn't produce a real ACORD 25 and is architecturally single-policy-per-cert. Options: retire it and repoint "New certificate," migrate its numbering/status/audit conventions into the new system, or run both in parallel during a transition. This needs an explicit decision — right now two "COI" concepts coexist un-reconciled and the UI points at the weaker one.
2. **Fixing the XFA-hybrid PDF rejection in `templateIngestion.ts`.** This isn't really optional — without it, a real ACORD 25 (or 125/126/140 later) likely can't even be onboarded as a template. Should be treated as a prerequisite, not a stretch goal.
3. **Cleanup of dead code** — the orphaned `pdf-generation-worker` edge function, the unwired `acord_generation_jobs` queue, and the `documents`-vs-`acord-forms` storage bucket naming mismatch. Recommend deciding whether to delete, repurpose, or formally deprecate these before the new module adds more surface area on top of an already-inconsistent base.
4. **How to represent "requested but not yet endorsed" Additional Insured status** (Decision 5) concretely in the UI and data model — a third state alongside yes/no, or a separate endorsement-reference field the user can leave blank. Also needs a decision on whether/how to extend `ai_type`/`waiver_of_subrogation` coverage to Auto and Property lines, which don't have it today (only GL/Umbrella do).
5. **Certificate Holder as a first-class concept.** Nothing in the schema models it today beyond a shared enum value on `policy_cgl_additional_interests`. The new Additional Insureds table effectively needs to become this concept — worth deciding whether `policy_cgl_additional_interests` should be migrated/merged into the new table or left alone (it also covers mortgagees, which are a different concept from certificate holders and shouldn't be conflated).
6. **Snapshot mechanics (Decision 4)** — precisely what gets frozen at issue time and where it's stored (a JSONB snapshot column on the new "issued COI" record, versus fully duplicating the row data) is left as an implementation detail for Fable's plan, informed by the reference doc's Section 4.3/4.4 trade-off discussion.
7. **Suggestions-table strategy for Additional Insureds (Section 3.3)** — new dedicated table (lower risk, some duplication) vs. generalizing `account_relationship_suggestions` polymorphically (less duplication, higher risk to the existing account-dedup feature). Recommend the dedicated-table path unless Fable sees a strong reason otherwise.

## 5. ACORD 25 domain mechanics carried forward from the reference doc

Brian's earlier file-based COI system (`ACORD_Module_Handoff.md`, attached) is offered there explicitly as one proven reference implementation, not a prescription — but its domain knowledge about ACORD 25 itself is hard-won and stack-independent. The parts still directly relevant to this repo:

- **Mental model**: an ACORD 25 = one **Master COI** (producer block, Named Insured, insurer table A–F, every in-force coverage line with limits/dates/policy numbers, optional remarks) + one **Certificate Holder** overlay. This is exactly Brian's "Master COI + Additional Insureds" split and should drive the data model 1:1.
- **Field-ID discipline**: ACORD field IDs are the contract between data model and PDF (`Subject_Attribute[_Qualifier]_Instance`, e.g. `Policy_GeneralLiability_EffectiveDate_A`). Never invent field names — extract them from the official blank PDF's AcroForm dictionary, the same way `templateIngestion.ts` already does for other forms (once its XFA rejection is fixed).
- **Insurer-letter mapping**: the 25 lists carriers once (A–F) and each coverage line references a carrier by letter. This is the one subtlety most likely to produce a "looks right but is wrong" bug — validate at render time that every coverage line's letter resolves to a populated insurer row, and that distinct carriers get distinct letters.
- **Checkbox values are field-specific names, not booleans** (e.g., `/1` for on, `/Off` for off, per-field) — pdf-lib's native `field.check()`/`uncheck()` (already used in `pdfFiller.ts`) handles this correctly as long as the field catalog captures the right export values per field, which is what `field_inventory`/`field_schema` on `acord_templates` are for.
- **Y/N code fields take literal `"Y"`/`"N"` strings**, never "Yes"/"No" — an easy validation rule to encode once and reuse everywhere ADDL INSD / SUBR WVD appear.
- **Never print premium on an ACORD 25** — limits only. This is a real, enforceable validation rule worth adding regardless of which engine renders the form.
- **Signature is a page overlay at fixed coordinates, not a form field** — the producer's printed name must match whoever's signature is stamped; keep this position map per form/edition since it can shift between ACORD editions.
- **Verification should be "rules as data," not hard-coded branches** — a per-form-type list of required fields plus a small rule engine, so ACORD 25's rule set (and later 125/126/140) is a config addition, not a new function. This maps naturally onto extending `acord_templates.validation_rules`, which already exists as a JSONB column in this repo's schema.
- **`NeedAppearances` vs. flatten-on-generate**: the reference doc's strongest warning ("filled ≠ visible") is already handled differently but validly in this repo via `updateFieldAppearances()` + `flatten()`. The risk isn't identical but the mitigation is the same: **visually verify the rendered PDF in a real browser after every generation**, not just confirm the fill call succeeded. Round-trip (fill → re-read fields) and visual (render page 1 to an image) tests should both exist for ACORD 25 specifically once it's onboarded.
- **Form edition drift**: pin the ACORD edition per template, store it with each generated cert, and treat the blank PDF as the source of truth to re-extract from whenever ACORD reissues the form — this maps directly onto `acord_templates.version`/`is_current`, which already exists.

## 6. What "done" looks like (restated for Fable's plan)

A staff user opens a customer record, sees a **Master COI** panel showing that customer's coverage picture pulled from their policies (carrier, policy number, dates, limits, NAIC, per-line Additional-Insured/waiver status with real endorsement tracking) in an easy-to-scan, easy-to-correct form. Separately, they maintain a shared **Additional Insureds** directory across the whole agency, with the system actively warning them when a new entry looks like one that already exists. When it's time to issue a certificate, they go to **ACORD Forms**, pick the customer, check the policies/coverage lines that belong on this cert, optionally add remarks, pick (or create) the Additional Insured who's the certificate holder, and click **Generate** — producing a real, correctly-filled, downloadable/printable ACORD 25 PDF that is automatically saved into that customer's Documents tab and preserved as an immutable historical record even if the underlying policy changes later. The whole flow should feel like the last, easy step after all the real work (keeping Master COI accurate) already happened — exactly as Brian described it.
